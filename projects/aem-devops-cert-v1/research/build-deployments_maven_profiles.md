# Maven Profiles for Environments

**Topic ID:** build-deployments.maven.profiles
**Researched:** 2026-03-29T00:00:00Z

## Overview

Maven profiles are named configuration sets that allow a single `pom.xml` to produce different build outputs depending on the target environment [1]. Instead of maintaining separate build files for local development, staging, and production, you define variations within one project and activate the appropriate profile at build time. This is central to AEM DevOps workflows, where the same project source must deploy to a local author instance, a local publish instance, a Cloud Manager pipeline, or a fully managed AEM as a Cloud Service environment [4].

Profiles can override or supplement nearly any section of the POM: dependencies, plugin configurations, resource directories, properties, and repository definitions. The activation mechanism is flexible — profiles can be triggered by a command-line flag, a JVM system property, an environment variable, a JDK version, an OS condition, or file presence/absence [1]. In AEM-specific projects generated by the archetype, a set of well-known profiles (`autoInstallPackage`, `autoInstallPackagePublish`, `classic`, etc.) provides the standard deploy-to-local workflow, while Cloud Manager relies on the presence or absence of the `CM_BUILD` environment variable to activate cloud-only build steps [4].

Resource filtering is the companion feature that completes the picture: once a profile sets environment-specific property values, resource filtering injects those values into configuration files (`.properties`, `.xml`, `.cfg.json`) at build time through `${placeholder}` substitution [2][3]. Together, profiles and resource filtering let a single source tree produce deployable artifacts correctly configured for each environment.

## Key Concepts

- **Profile definition location** — Profiles can live in three places: inside `pom.xml` (per-project, most portable), in `~/.m2/settings.xml` (per-user, not committed to source control), or in `${maven.home}/conf/settings.xml` (global, rarely used). Only `pom.xml` profiles can modify `<dependencies>`, `<plugins>`, and `<build>` sections [1].

- **Profile ID** — Every profile must have a unique `<id>` element. This ID is what you pass on the command line (`-Pdev`) or reference in IDE Maven tooling. The ID must be unique within the file where it is defined [1].

- **Explicit activation** — The most common method: `mvn clean install -Pdev` activates the `dev` profile. Multiple profiles can be comma-separated: `-Pdev,integrationTests`. Prefixing with `!` or `-` deactivates a profile: `-P!prod` [1].

- **Automatic activation** — Profiles activate implicitly based on conditions: a JDK version range, an OS family, the presence/value of a system property (`-Denv=prod`), an environment variable (`env.CM_BUILD`), or the existence/absence of a file. All conditions in a single `<activation>` block must be true simultaneously [1].

- **`activeByDefault`** — A profile can declare `<activeByDefault>true</activeByDefault>`. Crucially, this default is suppressed the moment any other profile in the same POM is explicitly activated — a common gotcha [1].

- **Resource filtering** — Enabled by adding `<filtering>true</filtering>` to a `<resource>` block. Maven replaces `${property.name}` tokens in text resources with values from POM properties, filter files, system properties, or CLI `-D` flags. Filtering is disabled by default to avoid corrupting binary resources [2][3].

- **Filter files** — External `.properties` files referenced via `<filters><filter>path/to/file.properties</filter></filters>`. A profile can point to a different filter file per environment, keeping secrets out of `pom.xml` [2].

- **`CM_BUILD` environment variable** — Adobe Cloud Manager sets this variable during pipeline builds. AEM projects use it to gate cloud-only build steps: activate a profile when `env.CM_BUILD` is present, or deactivate local-only profiles when it is absent [4].

- **AEM archetype deploy profiles** — The standard AEM Project Archetype generates four deploy profiles: `autoInstallPackage` (author, port 4502), `autoInstallPackagePublish` (publish, port 4503), `autoInstallBundle` (OSGi bundle only), and `integrationTests`. These are not environment profiles in the classical sense — they control the deployment target, not environment-specific configuration [5].

- **Variable substitution via `-D` flags** — Profile default properties (e.g., `aem.host=localhost`, `aem.port=4502`) can be overridden at build time with `-Daem.host=prod.hostname -Daem.port=4502`. This pattern allows the same profile to deploy to different hosts without editing `pom.xml` [5].

## Technical Details

### Profile Structure in pom.xml

The following example demonstrates environment profiles with automatic property activation and resource filtering [1][2]:

```xml
<profiles>
  <!-- Default: local development -->
  <profile>
    <id>dev</id>
    <activation>
      <activeByDefault>true</activeByDefault>
    </activation>
    <properties>
      <db.url>jdbc:mysql://localhost:3306/dev_db</db.url>
      <db.username>dev_user</db.username>
      <sling.url>http://localhost:4502/crx/packmgr/service.jsp</sling.url>
    </properties>
  </profile>

  <!-- Stage environment, activated by system property -->
  <profile>
    <id>stage</id>
    <activation>
      <property>
        <name>env</name>
        <value>stage</value>
      </property>
    </activation>
    <properties>
      <db.url>jdbc:mysql://stage-db:3306/stage_db</db.url>
      <db.username>stage_user</db.username>
    </properties>
  </profile>

  <!-- Production environment -->
  <profile>
    <id>prod</id>
    <properties>
      <db.url>jdbc:oracle:thin:@prod-server:1521:PROD</db.url>
      <db.username>prod_user</db.username>
    </properties>
  </profile>
</profiles>
```

Activate the stage profile: `mvn clean install -Denv=stage` [1].

### Resource Filtering Setup

Enable filtering so that `${property.name}` tokens in resource files get replaced at build time [2][3]:

```xml
<build>
  <resources>
    <resource>
      <directory>src/main/resources</directory>
      <filtering>true</filtering>
      <!-- Only filter text files, never binary files -->
      <includes>
        <include>**/*.properties</include>
        <include>**/*.xml</include>
      </includes>
    </resource>
    <!-- Binary resources: filtering disabled -->
    <resource>
      <directory>src/main/resources</directory>
      <filtering>false</filtering>
      <excludes>
        <exclude>**/*.properties</exclude>
        <exclude>**/*.xml</exclude>
      </excludes>
    </resource>
  </resources>
</build>
```

A resource file `src/main/resources/datasource.properties` can then reference profile-defined values:

```properties
db.url=${db.url}
db.username=${db.username}
```

### Cloud Manager Profile Pattern

This is the canonical AEM pattern for Cloud Manager builds [4]:

```xml
<!-- Cloud Manager only: activated by CM_BUILD env var -->
<profile>
  <id>cmBuild</id>
  <activation>
    <property>
      <name>env.CM_BUILD</name>
    </property>
  </activation>
  <!-- Cloud-specific build steps, package repositories, etc. -->
</profile>

<!-- Local development only: activated by ABSENCE of CM_BUILD -->
<profile>
  <id>localDev</id>
  <activation>
    <property>
      <name>!env.CM_BUILD</name>
    </property>
  </activation>
  <!-- Local install, test steps, etc. -->
</profile>
```

To simulate a Cloud Manager build locally: `mvn clean install -PcmBuild` [4].

### AEM Archetype Deploy Profiles

Standard commands from the AEM Project Archetype [5]:

```bash
# Deploy all packages to local Author (port 4502)
mvn clean install -PautoInstallPackage

# Deploy to local Publish (port 4503)
mvn clean install -PautoInstallPackagePublish

# Deploy only the OSGi bundle to Author
mvn clean install -PautoInstallBundle

# Deploy to a remote author with custom credentials
mvn clean install -PautoInstallPackage -Daem.host=prod.example.com -Dsling.password=secret

# AEM 6.5 / 6.4 (on-premise): append the classic profile
mvn clean install -PautoInstallSinglePackage -Pclassic
```

### Checking Active Profiles

```bash
# List all active profiles for the current build
mvn help:active-profiles

# With a property trigger
mvn help:active-profiles -Denv=stage

# See the effective POM after profile merging
mvn help:effective-pom -Pprod -Doutput=effective-pom.xml
```

## Common Patterns

**Pattern 1: Properties-only profiles with resource filtering.** Each profile defines a `<properties>` block with environment-specific values. A shared resource file uses `${property.name}` tokens. Filtering is enabled in `<build><resources>`. This is the simplest and most portable approach [1][2].

**Pattern 2: External filter files per profile.** Each profile points to a different `.properties` file via `<build><filters>`. Useful when you want to keep environment credentials out of `pom.xml` and in files that can be excluded from version control via `.gitignore` [2].

```xml
<profile>
  <id>prod</id>
  <build>
    <filters>
      <filter>src/main/filters/prod.properties</filter>
    </filters>
  </build>
</profile>
```

**Pattern 3: AEM Cloud Manager gate.** Check `env.CM_BUILD` to run steps only inside the pipeline (e.g., specific code quality plugins, artifact signing) or only on developer machines (e.g., auto-install, integration tests against local AEM) [4].

**Pattern 4: Classifier-tagged artifacts.** Build environment-specific JARs or content packages with `<classifier>` set per profile (e.g., `foo-1.0-dev.jar`, `foo-1.0-prod.jar`). Useful when artifacts must be stored or promoted by environment [1].

**Pattern 5: Combine profiles.** A single `mvn` invocation can combine a deploy profile and an environment profile: `mvn clean install -PautoInstallPackage -Pstage`. The profiles' configurations merge, with later profiles overriding earlier ones when the same property is defined in both [1].

## Gotchas

**`activeByDefault` is suppressed by any explicit activation.** If the `dev` profile has `<activeByDefault>true</activeByDefault>` and you run `mvn install -Pprod`, the `dev` profile is NOT active. This surprises developers who expect the default to always be included [1].

**Profile conditions are AND, not OR.** If you specify multiple conditions inside a single `<activation>` block (e.g., a JDK version AND a property), all conditions must be satisfied. To create OR logic, define separate profiles with the same configuration [1].

**Filtering binary files corrupts them.** Maven's resource filtering performs string replacement on the raw bytes. Any binary file (image, PDF, keystore, font) that goes through `<filtering>true</filtering>` will likely be corrupted. Always use include/exclude lists or a separate `src/main/resources-filtered` directory for text-only files [3].

**Profiles in `settings.xml` cannot configure dependencies or plugins.** External profiles (per-user or global `settings.xml`) are restricted to `<repositories>`, `<pluginRepositories>`, and `<properties>`. They cannot add dependencies or plugin configuration. Only `pom.xml`-defined profiles have full access to the build configuration [1].

**`mvn clean` required when switching profiles between builds.** The `target/classes` directory retains files from the previous build. Running `mvn install -Pdev` followed by `mvn install -Pprod` without a `clean` may result in a mixed-environment artifact. Always run `mvn clean install -Pprofile` [1].

**`autoInstallPackage` vs `autoInstallSinglePackage`.** In older archetype versions, `autoInstallPackage` deploys individual submodule packages. In newer versions, `autoInstallSinglePackage` deploys the aggregate `all` package. Using the wrong profile can result in incomplete deployments. Check which profile names your specific archetype version generates [5].

**The `classic` profile is required for AEM 6.5/6.4.** Projects generated by recent archetype versions default to AEM as a Cloud Service behavior. Without `-Pclassic`, building against AEM 6.5 may fail or produce incorrect packages. This must be explicitly added to every `mvn` invocation when targeting on-premise AEM [5].

**`CM_BUILD` is set by Cloud Manager, not by AEM itself.** If you're running a Jenkins or GitHub Actions pipeline outside Cloud Manager, `CM_BUILD` will NOT be set. Profiles relying on it for cloud-only steps will not activate. You must set the variable explicitly in other CI environments: `export CM_BUILD=true` [4].

**Variable resolution order matters.** The priority for `${property}` resolution in resource filtering is: (1) CLI `-D` flags, (2) POM `<properties>`, (3) system properties, (4) filter files. A CLI flag always wins, which enables overriding profile defaults without editing the POM [2].
