# Frontend Pipeline Configuration

**Topic ID:** build-deployments.frontend.pipeline-config
**Researched:** 2026-03-29T00:00:00Z

## Overview

The AEM Cloud Manager frontend pipeline is a specialized CI/CD pipeline type that deploys only CSS, JavaScript, and static assets — independent of AEM's full-stack Maven build. Rather than compiling and packaging Java code, OSGi bundles, and content packages, the frontend pipeline runs `npm run build` against the `ui.frontend` module and deploys the resulting `dist/` folder directly to AEM's built-in CDN [1][2]. This separation is the defining characteristic: frontend deployments are decoupled from backend deployment cycles, enabling front-end teams to iterate on themes and styles without triggering full application rebuilds.

The pipeline was introduced to address a core pain point: in the traditional full-stack model, any CSS or JavaScript change required a complete Maven build, OSGi deployment, and Sling content replication — a process that could take 30+ minutes. The frontend pipeline compresses that to minutes because it only handles static file compilation and CDN upload [2]. Assets are served via a static CDN hostname (e.g., `https://static-...`), not through AEM's `/etc.clientlibs/` path, which also improves cache efficiency [3].

In practice, the frontend pipeline is most valuable for sites with active theming work or a dedicated front-end team. It introduces a contract boundary between front-end and back-end teams: the HTML/JSON output rendered by AEM becomes the agreed-upon interface, and both teams must coordinate changes carefully when that contract changes [1].

## Key Concepts

- **Pipeline type: Front End Code** — When creating a pipeline in Cloud Manager, selecting "Front End Code" designates it as a frontend pipeline. This is distinct from Full-Stack, Web Tier Config, and Config pipeline types [4].

- **Build command: `npm run build`** — Cloud Manager always runs `npm run build` for frontend pipelines. The project's `package.json` must define this script. No other build command can be substituted [1][2].

- **Output: `dist/` folder** — The contents of the `dist/` directory after `npm run build` are what Cloud Manager deploys. If the build outputs to a different folder (e.g., `build/`), the pipeline will fail silently or produce an empty deployment [3].

- **CDN delivery** — Deployed assets are hosted on Adobe's CDN and served from a `https://static-` domain, not from AEM's publish tier. AEM pages reference these assets via `<link>` and `<script>` tags generated by the Page Component [1][4].

- **Sling config handshake** — Enabling the frontend pipeline creates `SiteConfig` and `HtmlPageItemsConfig` nodes under `/conf/<site-name>/sling:configs`. The `prefixPath` property in `HtmlPageItemsConfig` stores the CDN URL prefix for frontend assets [3].

- **Core Components v2 prerequisite** — The site's Page Component must be v2 or newer from AEM Core Components. This component generates the `<link>`/`<script>` tags that reference CDN-hosted frontend assets [2][5].

- **Node.js version control** — The `NODE_VERSION` environment variable on the pipeline controls the Node.js version used during build. Supported versions: 12, 14 (default), 16, 18, 20, 22, 23 [1].

- **Code Location field** — When configuring the pipeline in Cloud Manager, the "Code Location" field must be set to the path of the frontend module (typically `/ui.frontend`). This scopes what the pipeline checks out and builds [3][4].

- **Multiple concurrent pipelines** — Unlike full-stack pipelines (one per environment), multiple frontend pipelines can run concurrently against the same program — up to 300 total pipelines per program [4].

- **IP Allow List requirement** — Cloud Manager's IP Allow List must be configured before enabling the frontend pipeline, as the build process makes requests to the AEM author for proxy-based local development [5].

## Technical Details

### Pipeline Setup in Cloud Manager

From the Pipelines section in Cloud Manager:

1. Click **Add** and select **Add Non-Production Pipeline** (for dev/stage) or **Add Production Pipeline**.
2. In the Configuration tab, select **Deployment Pipeline**, provide a name (e.g., `wknd-theme-prod`).
3. In the Source Code tab:
   - **Pipeline Type**: Select **Front End Code**
   - **Environment**: Choose the target from Eligible Deployment Environments
   - **Repository** and **Git Branch**: Point to the correct branch
   - **Code Location**: Set to `/ui.frontend` (or the relevant module path)
4. Save and run [3][4].

### Enabling the Pipeline on the AEM Site Side

Before the frontend pipeline can function, the AEM site must be configured to use it:

1. In AEM Sites console, select the site root.
2. Open the **Site** rail from the rail selector.
3. Click **Enable Front End Pipeline**.
4. Confirm changes — AEM creates the Sling config nodes and adds empty CSS/JS include references to all pages [2][5].

This button only appears when the Page Component is Core Components v2+. The initial CSS/JS files deployed are empty; developers then download the "theme sources" folder and begin development [2].

### The Frontend-Backend Contract

The contract model is critical [1]:

```
AEM (backend) renders HTML/JSON
        ↕  ← Contract boundary →
Frontend pipeline deploys CSS/JS that styles that HTML
```

When the contract must change (e.g., new HTML markup from a backend component change):

1. Backend deploys new HTML rendering to dev environment.
2. Frontend uses `npx aem-site-theme-builder proxy` with `AEM_URL` env var to test CSS/JS against both old and new HTML locally.
3. Backend deploys to production.
4. Frontend finalizes CSS/JS and deploys via frontend pipeline.

### Local Proxy Development

```bash
# Point local theme dev to an AEM environment
AEM_URL=https://author-p12345-e67890.adobeaemcloud.com npx aem-site-theme-builder proxy
```

This retrieves content from the AEM environment but serves frontend files from the local `dist/` folder, enabling rapid iteration without full pipeline runs [1].

### Migration from Full-Stack to Frontend Pipeline

When converting an existing project to use the frontend pipeline [3]:

1. Remove `ui.frontend` from the Maven modules in the root `pom.xml` (it should no longer build as part of the full-stack build).
2. Remove `clientlib.config.js` — it is only needed when copying build output to `ui.apps` for clientlib packaging.
3. Update the `ui.frontend` module to output to `dist/` and ensure `package.json` has a `build` script.
4. Run the full-stack pipeline first to remove old clientlib files and establish the Sling config.
5. Then run the frontend pipeline to deploy assets to CDN.

### Pipeline Type Comparison

| Attribute | Full-Stack | Front-End | Web Tier Config | Config |
|---|---|---|---|---|
| Deploys | Java + CSS/JS + Dispatcher | CSS/JS only | HTTPD/Dispatcher | CDN/WAF/logs |
| Speed | Slow (30+ min) | Fast (minutes) | Fast | Fast |
| Count per env | 1 | Multiple | 1 | 1 |
| Code scanning | Yes | No | No | No |
| UI testing | Yes | No | No | No |
| Min AEM version | Any | Any | 2021.12.6151+ | Any |

## Common Patterns

**Pattern 1: Parallel team development** — Backend team uses the full-stack pipeline for Java/OSGi/Dispatcher changes. Frontend team uses a dedicated frontend pipeline for CSS/JS theme changes. Teams coordinate on the contract (HTML structure). This is the primary use case for the frontend pipeline [1][2].

**Pattern 2: Quick site theming** — For sites created using the AEM Quick Site Creation wizard, the frontend pipeline is the default and expected workflow. Designers download the theme sources ZIP, customize CSS/JS, commit to a branch, and the frontend pipeline deploys to CDN [2].

**Pattern 3: Multiple front-end pipelines per program** — One pipeline per site or per environment (e.g., `wknd-theme-dev`, `wknd-theme-prod`). Naming convention should include the module name (from `package.json`) and the target environment to avoid conflicts [1].

**Pattern 4: Rollback via re-run** — Because the pipeline deploys to CDN, rolling back is done by re-running the pipeline from a previous Git commit. There is no "undo" button in Cloud Manager for frontend deployments.

## Gotchas

**1. `dist/` vs `build/` output folder** — Cloud Manager expects `dist/` as the output directory. If your `npm run build` writes to `build/` or any other folder, the pipeline will deploy nothing or fail. This is the most common silent failure in new frontend pipeline setups [3].

**2. Run full-stack first during migration** — When migrating from full-stack to frontend pipeline, you must run the full-stack pipeline first (after modifying it to remove `ui.frontend` from the Maven build). This run removes old clientlib files from AEM and creates the Sling config nodes. Running the frontend pipeline before this step will result in the CDN assets not being referenced by AEM pages [3].

**3. Existing clientlibs are NOT auto-converted** — Enabling the frontend pipeline does not migrate your existing `/etc.clientlibs/` resources. Manual migration by a frontend developer is required. Until migration is complete, both old clientlibs and new CDN assets may coexist, potentially causing style conflicts [2][5].

**4. Only Full-Stack supports testing gates** — Code scanning, function testing, UI testing, and experience audit are only available in the Full-Stack pipeline. Frontend pipelines have no built-in quality gates beyond a successful `npm run build` [4]. This is a common exam question mismatch: teams expecting to run Selenium/Playwright via the frontend pipeline will find no such option.

**5. Node.js version gotcha** — The default Node.js version in Cloud Manager build agents is 14. If your project uses modern JS features or dependencies requiring Node 18/20, set `NODE_VERSION=20` as a build environment variable on the pipeline. Node 19/20/21 may have LMDB compatibility issues with some AEM theme dependencies [1][5].

**6. One full-stack pipeline per environment** — You cannot have two full-stack pipelines deploying to the same environment simultaneously. Frontend pipelines do not have this restriction, but two frontend pipelines must not deploy to the same site/environment concurrently to avoid race conditions [4][1].

**7. IP Allow List must be configured before enabling** — If Cloud Manager's IP Allow List feature is in use, it must be updated to allow the AEM author's IP before clicking "Enable Front End Pipeline." Forgetting this blocks the proxy-based development workflow [5].

**8. Core Components v2 is a hard prerequisite** — Sites built on older Page Component versions cannot use the frontend pipeline without upgrading Core Components. The pipeline's CDN URL injection relies on the Page Component emitting `<link>` and `<script>` tags configured by the Sling config nodes [2][5].

**9. Parallel pipeline conflict** — Multiple frontend pipelines can coexist per program, but two pipelines targeting the same site + environment must not run simultaneously. Adobe recommends systematic naming to track which pipeline owns which site/environment pair [1].
