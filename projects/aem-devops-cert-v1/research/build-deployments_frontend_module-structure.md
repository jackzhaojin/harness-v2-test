# Frontend Module Structure

**Topic ID:** build-deployments.frontend.module-structure
**Researched:** 2026-03-29T00:00:00Z

## Overview

The `ui.frontend` module is the central location for all front-end resources (JavaScript, CSS, images, fonts) in AEM projects generated by the AEM Project Archetype [1]. Introduced with Archetype 20, it replaced ad-hoc clientlib organization by wrapping a Webpack-based build system around front-end source code, enabling modern tooling like TypeScript, Sass/SCSS, tree-shaking, and autoprefixing [2]. Source code lives in `ui.frontend/src/` and is compiled into optimized bundles by Webpack, which the `aem-clientlib-generator` then packages into AEM client libraries (clientlibs) inside the `ui.apps` module [3].

The build and deployment flow bridges modern front-end workflows with AEM's server-side asset management. During development, `npm run start` spins up a Webpack dev server on port 8080 with a proxy to a local AEM instance at `localhost:4502`, allowing front-end developers to work with near-live reloading without running a full Maven build [4]. In CI/CD pipelines, the `frontend-maven-plugin` in `ui.frontend/pom.xml` triggers the npm build scripts automatically as part of the full Maven build, ensuring clientlibs are regenerated and synced to `ui.apps` before AEM package deployment [2].

There are two distinct deployment paths for the `ui.frontend` output: the **full-stack pipeline** (traditional approach, bundles front-end assets as AEM clientlibs, delivered via `/etc.clientlibs/` URLs) and the **front-end pipeline** (AEM as a Cloud Service feature that deploys front-end assets directly to the built-in CDN, bypassing clientlibs) [5]. Understanding the differences between these two paths — and when each applies — is a key exam topic.

## Key Concepts

- **`ui.frontend` module** — A Maven sub-module in the AEM project archetype that acts as the self-contained front-end build context. All JS, CSS, fonts, images, and SCSS source files live here under `src/` [1][2].

- **Webpack entry point** — The single entry file that Webpack uses to resolve the entire dependency graph. In the standard archetype, this is `src/main/webpack/site/main.ts` (TypeScript) or `src/main/webpack/site/index.js` (JS), from which all component JS and SCSS files are imported [3][4].

- **Three-file Webpack configuration** — The build uses `webpack.common.js` (shared rules), `webpack.dev.js` (source maps, dev server, AEM proxy), and `webpack.prod.js` (minification via TerserPlugin, CSS minimization via CssMinimizerPlugin) [1][2].

- **`aem-clientlib-generator`** — An npm plugin that takes already-compiled CSS/JS bundles from the `dist/` directory and transforms them into the AEM clientlib format (generating `js.txt`, `css.txt`, `js/`, `css/`, `resources/` folders) [6]. It does NOT compile code — it only packages already-built assets.

- **`clientlib.config.js`** — Configuration file for `aem-clientlib-generator` that defines where clientlibs are written, their categories, `allowProxy` flag, embedded/dependency relationships, and asset glob patterns [6].

- **Two generated clientlibs** — The standard archetype output produces `clientlib-site` (site-specific compiled CSS/JS) and `clientlib-dependencies` (third-party dependency bundles), both written to `ui.apps/src/main/content/jcr_root/apps/<appId>/clientlibs/` [2][3].

- **`allowProxy` property** — A critical setting in clientlib configuration and in the generated `.content.xml`. When `true`, AEM serves the clientlib from the proxy path `/etc.clientlibs/` instead of the actual `/apps/` path, which is required because Dispatcher typically blocks direct access to `/apps/` [7].

- **Full-stack pipeline vs. front-end pipeline** — Full-stack deploys clientlibs via `/etc.clientlibs/`; the front-end pipeline deploys static files directly to AEM's built-in CDN with URLs like `https://static-pXX-eYY.p123-e456.adobeaemcloud.com/...` [5].

- **`frontend-maven-plugin`** — Maven plugin configured in `ui.frontend/pom.xml` that triggers `npm install` and `npm run prod` during a Maven build, orchestrating Webpack bundling and clientlib generation automatically [2].

## Technical Details

### Source Directory Structure

The `ui.frontend` module follows this folder hierarchy [3][4]:

```
ui.frontend/
├── src/
│   └── main/
│       └── webpack/
│           ├── site/
│           │   ├── main.ts         ← Webpack entry point
│           │   └── main.scss       ← Primary stylesheet
│           ├── components/
│           │   └── <component>/    ← Component JS + SCSS files
│           ├── resources/          ← Static assets (fonts, icons)
│           └── static/             ← Static HTML for dev server
├── dist/                           ← Generated output (gitignored)
│   ├── clientlib-site/
│   │   ├── site.js
│   │   ├── site.css
│   │   └── resources/
│   └── clientlib-dependencies/
│       ├── dependencies.js
│       └── dependencies.css
├── webpack.common.js
├── webpack.dev.js
├── webpack.prod.js
├── clientlib.config.js
├── package.json
├── tsconfig.json
└── pom.xml
```

### Webpack Configuration

`webpack.dev.js` extends `webpack.common.js` and adds [1][2]:
- Source maps enabled (`devtool: 'eval-source-map'`)
- Webpack Dev Server on `localhost:8080`
- Proxy to AEM at `localhost:4502`
- JS and CSS optimization **disabled** for faster iteration

`webpack.prod.js` extends `webpack.common.js` and adds [1][2]:
- TerserPlugin for JS minification
- CssMinimizerPlugin for CSS compression
- Tree-shaking (dead code elimination)
- Source maps **disabled**
- CSS autoprefixing via PostCSS

The output `property` in `webpack.common.js` directs the bundle to `clientlib-site/js/[name].bundle.js` in the `dist` folder [2].

### clientlib.config.js Example

The following pattern is from the official archetype documentation [6]:

```js
module.exports = {
  context: __dirname,
  clientLibRoot: "./../ui.apps/src/main/content/jcr_root/apps/my-site/clientlibs",
  libs: [
    {
      name: "clientlib-site",
      allowProxy: true,
      categories: ["my-site.site"],
      dependencies: ["my-site.dependencies"],
      serializationFormat: "xml",
      cssProcessor: ["default:none", "min:none"],
      jsProcessor: ["default:none", "min:gcc;compilationLevel=whitespace"],
      assets: {
        js: {
          cwd: "dist/clientlib-site",
          files: ["**/*.js"],
          flatten: false
        },
        css: {
          cwd: "dist/clientlib-site",
          files: ["**/*.css"],
          flatten: false
        },
        resources: {
          cwd: "dist/clientlib-site",
          files: ["**/*.*"],
          flatten: false,
          ignore: ["**/*.js", "**/*.css"]
        }
      }
    }
  ]
};
```

### Generated Clientlib Anatomy

Each generated clientlib directory contains [2][6]:
- `js/` — JavaScript files
- `css/` — CSS files
- `resources/` — Source maps, code chunks from code splitting, static assets
- `js.txt` — Ordered list of JS files for AEM to merge/serve
- `css.txt` — Ordered list of CSS files for AEM to merge/serve
- `.content.xml` — JCR node definition with `categories`, `allowProxy`, `dependencies`

### NPM Scripts

| Script | Behavior |
|--------|----------|
| `npm run start` | Launches Webpack dev server on `localhost:8080`, proxies API calls to `localhost:4502` |
| `npm run dev` | Full build, optimization OFF, source maps ON — outputs to `dist/` then generates clientlibs |
| `npm run prod` | Full build, optimization ON (tree-shaking, minification), source maps OFF — production-ready |

## Common Patterns

**Component-level JS/CSS organization.** The recommended pattern is to colocate each component's JS and SCSS files under `ui.frontend/src/main/webpack/components/<component-name>/`. No `js.txt`, `css.txt`, or `.content.xml` files are needed at the component level — Webpack's globber automatically pulls in all JS/CSS files under `/components/` via the entry point import [2].

**Globbing instead of explicit imports.** The standard archetype entry `main.ts` uses glob patterns (via `require.context` or similar) to auto-include all component-level JS/CSS. This means developers can add a new component's `.ts`/`.scss` files and they're picked up automatically — no manual import registration [2].

**Separation of site CSS from dependency CSS.** Third-party library CSS (e.g., select2.css) belongs in `clientlib-dependencies`, while site-specific styles go in `clientlib-site`. This separation enables the correct load ordering and allows dependency libraries to be updated or swapped without touching brand CSS [1][3].

**Including clientlibs in page templates.** The two generated clientlibs are included in AEM page templates via template policy configuration. `clientlib-dependencies` typically loads in `<head>` while `clientlib-site` loads before `</body>` (or in `<head>` if explicitly configured) [7].

**Maven integration.** Running `mvn clean install -PautoInstallSinglePackage` from the project root triggers the complete chain: `frontend-maven-plugin` runs `npm install` then `npm run prod`, Webpack compiles to `dist/`, `aem-clientlib-generator` writes to `ui.apps/`, Maven packages `ui.apps` and deploys it to AEM [2].

**Front-end pipeline requirement.** When enabling the AEM as a Cloud Service front-end pipeline, the `ui.frontend/package.json` must have a `build` script (`npm run build`), and Cloud Manager executes `npm run build` to produce the deployable `dist/` folder. In this mode, there is no clientlib generation — the `dist/` contents are deployed as static CDN assets [5].

## Gotchas

**`aem-clientlib-generator` does not compile — it only packages.** A common misconception is that it handles build compilation. It only takes already-compiled `dist/` output and converts it to the clientlib JCR structure. If Webpack fails, clientlib generation will use stale or empty output [6].

**Webpack Hot Module Replacement (HMR) is disabled.** The standard archetype configuration does NOT support HMR. Changes require a full recompile, clientlib regeneration, AEM sync, and browser refresh. The dev server at `localhost:8080` provides live static previewing but changes aren't reflected in AEM automatically without `npm run watch` plus `aemsync` [2].

**Full-stack vs. front-end pipeline: clientlibs ARE still generated for full-stack.** Switching to the front-end pipeline removes the need for clientlib generation in Cloud Manager deployments, but local AEM SDK development still requires clientlib generation via `npm run dev`. Teams must maintain two paths [5].

**`allowProxy` must be `true` for production security.** If `allowProxy` is `false` or missing, clientlibs are served from `/apps/...` paths, which Dispatcher typically blocks. The canonical delivery path is `/etc.clientlibs/<app>/clientlibs/clientlib-site.min.css` [7].

**`serializationFormat: "xml"` vs default `"json"`.** The `aem-clientlib-generator` supports two output formats. The `xml` format generates FileVault-compatible `.content.xml` files needed for standard AEM package deployment. The `json` format is simpler but less common in Maven-based full-stack projects. Mismatching the format causes the generated clientlib structure to be incompatible with the expected package installation [6].

**`clientlib-dependencies` vs `clientlib-base` confusion.** The archetype generates `clientlib-site` and `clientlib-dependencies` from `ui.frontend`. Separate from these, the WKND reference project also defines `clientlib-base` and `clientlib-grid` within `ui.apps` directly (not from `ui.frontend`). `clientlib-base` embeds Core Component libraries; `clientlib-grid` provides responsive layout CSS. These are NOT outputs of the `ui.frontend` webpack build [7].

**Category naming collision.** The clientlib categories defined in `clientlib.config.js` (e.g., `my-site.site`, `my-site.dependencies`) must match what is specified in the page template policy. A mismatch means the clientlib is never included in rendered pages — a subtle issue that doesn't produce build errors but results in missing styles/scripts in the browser [6][7].

**Front-end pipeline initial switchover causes unstyled pages.** When migrating from full-stack to front-end pipeline, there is a gap period after the full-stack pipeline removes old clientlibs but before the front-end pipeline deploys CDN assets. Teams should plan for temporary unstyled pages during this one-time transition [5].
