# Evaluation Tool Features

**Topic ID:** evaluation.console-tool.features
**Researched:** 2026-03-01T12:00:00Z

## Overview

The Anthropic Console Evaluation tool is a developer-focused interface within platform.claude.com that enables systematic testing, comparison, and refinement of prompts before production deployment [1]. Unlike the conversational claude.ai interface, this workspace is designed for creating reusable, production-quality prompts through data-driven iteration [2].

The tool addresses a fundamental challenge in AI development: prompt quality significantly impacts results, but crafting effective prompts requires both deep domain expertise and knowledge of large language model behavior [2]. By integrating test case generation, side-by-side output comparison, quality grading, and prompt versioning into a single workflow, the Console democratizes high-quality prompt engineering and enables teams to iterate systematically rather than relying on subjective assessment alone [2][3].

## Key Concepts

- **Evaluation Tool** — A Console feature that tests prompts against multiple scenarios with automatic test case generation, output comparison, and quality grading [1]. Accessible via the "Evaluate" tab in the prompt editor.

- **Dynamic Variables** — Template placeholders using double brace syntax `{{variable}}` that allow test inputs to be injected into prompts [1]. Required for creating evaluation test sets; the prompt generator automatically creates prompts with this syntax.

- **Test Cases** — Input-output pairs used to verify prompt behavior across scenarios [1]. Can be created manually, imported via CSV, or auto-generated by Claude [1][2].

- **Side-by-Side Comparison** — Feature allowing developers to view outputs from two or more prompt versions simultaneously to understand how changes affect results [1][2].

- **5-Point Quality Grading** — Structured scoring system where subject matter experts rate response quality from 1 (poor) to 5 (excellent) to track improvements across prompt iterations [1][4].

- **Prompt Versioning** — Capability to create new versions of a prompt and re-run the entire test suite against each version for rapid iteration [1][2].

- **Ideal Output Column** — Optional reference column in evaluations allowing users to define perfect responses for benchmarking and consistent grading [5].

- **Prompt Improver** — Automated refinement feature that applies advanced techniques (chain-of-thought, XML formatting) to enhance prompt quality [5].

## Technical Details

### Accessing the Evaluation Tool

To use the Evaluation tool [1]:
1. Open the Claude Console at platform.claude.com
2. Navigate to the prompt editor and compose your prompt
3. Click the "Evaluate" tab at the top of the screen
4. Ensure your prompt includes at least 1-2 dynamic variables using `{{variable}}` syntax

### Test Case Creation Methods

The Console supports three approaches for building test suites [1]:

```
Manual Entry:     Click "+ Add Row" to add individual test cases
CSV Import:       Upload structured test data with headers matching variable names
Auto-Generation:  Click "Generate Test Case" for Claude to create cases automatically
```

For auto-generation, Claude creates test cases one row at a time per click. The generation logic can be customized by clicking the dropdown arrow next to "Generate Test Case" and selecting "Show generation logic" to fine-tune what types of cases Claude generates [1].

### Prompt Structure for Evaluation

Effective evaluation requires prompts with clear input/output formatting [1]:

```text
In this task, you will generate a cute one sentence story that incorporates
two elements: a color and a sound.
The color to include in the story is:
<color>
{{COLOR}}
</color>
The sound to include in the story is:
<sound>
{{SOUND}}
</sound>
```

This structure makes it easy to vary inputs (`{{COLOR}}` and `{{SOUND}}`) and evaluate outputs consistently across test cases.

### 5-Point Grading Scale Criteria

The standard rubric for quality grading follows this structure [4]:

| Score | Description |
|-------|-------------|
| 5 (Excellent) | Accurate, complete, clear, helpful tone |
| 4 (Good) | Accurate and complete, but could be clearer |
| 3 (Acceptable) | Mostly accurate, missing some details |
| 2 (Needs Work) | Partially correct, significant gaps |
| 1 (Poor) | Incorrect or misleading information |

### Prompt Improver Techniques

The prompt improver automatically refines prompts using five methods [5]:

1. **Chain-of-thought reasoning** — Adds a dedicated section for Claude to think through problems systematically before responding
2. **Example standardization** — Converts examples into consistent XML format for improved clarity
3. **Example enrichment** — Augments existing examples with chain-of-thought reasoning
4. **Rewriting** — Clarifies structure and corrects grammatical issues
5. **Prefill addition** — Prefills Assistant messages to direct actions and enforce output formats

### Grading Methods in Evaluations

Beyond manual 5-point grading, evaluations support three grading approaches [6]:

| Method | Speed | Cost | Best For |
|--------|-------|------|----------|
| Code-based | Fastest | Cheapest | Deterministic outputs (exact match, regex) |
| Model-based | Fast | Low | Subjective tasks, open-ended responses |
| Human | Slow | High | Complex judgment calls, final validation |

## Common Patterns

### Standard Evaluation Workflow

A typical evaluation workflow progresses through these stages [3]:

1. **Define Criteria** — Establish 3-5 scoring dimensions (accuracy, tone, completeness) before testing
2. **Design Test Cases** — Create 15-20 cases across four categories: straightforward, boundary, complex, and adversarial
3. **Create Golden Dataset** — Write ideal responses for key test cases to serve as benchmarks
4. **Import and Score** — Run tests and score responses on your rubric for each dimension
5. **Iterate** — Identify weakest performance area, modify prompt, re-run relevant test subset

### Four Test Case Categories

Comprehensive test suites should cover [3]:

- **Straightforward**: Basic factual questions testing core functionality
- **Boundary**: Out-of-scope queries requiring graceful deflection
- **Complex**: Questions requiring synthesis across multiple contexts
- **Adversarial**: Challenging or pressure-based scenarios that stress-test robustness

### Production Iteration Pattern

When improving prompts for production [1][2]:
1. Run initial evaluation across all test cases
2. Compare outputs side-by-side to identify patterns
3. Create a new prompt version with targeted improvements
4. Re-run the entire eval suite against the new version
5. Grade responses on the 5-point scale
6. Repeat until quality targets are met

## Gotchas

- **Variable Syntax Required** — The Evaluation tool requires at least 1-2 dynamic variables using `{{variable}}` syntax. Prompts without variables cannot create eval test sets [1]. This is a common stumbling block for developers new to the Console.

- **CSV Headers Must Match Exactly** — When importing test cases via CSV, column headers must match template variable names precisely (case-sensitive) [3]. Mismatched headers cause silent failures where variables are not populated.

- **Grading is Ongoing Cost** — Every time you re-run an eval, you incur grading costs. Since evaluations are run repeatedly during development, structure questions to enable automated grading where possible rather than relying entirely on human grading [6].

- **Model-Based Graders Need Validation** — The effectiveness of model-based grading varies by task. Always test grader accuracy on a sample of outputs before trusting automated scores [6].

- **Prompt Improver Changes May Break Tests** — The prompt improver rewrites prompts substantially (adding chain-of-thought sections, reformatting examples). After improvement, re-run all test cases because output format changes may cause previously passing tests to fail [5].

- **Side-by-Side Comparison vs. Versioning** — Side-by-side comparison shows outputs from different prompt versions simultaneously, but versioning tracks the prompt text history. Both are distinct features that work together—do not confuse viewing outputs (comparison) with storing prompt iterations (versioning) [1][2].

- **Ideal Output is Optional but Valuable** — The ideal output column is optional, but defining golden answers forces product decisions about length, tone, and information hierarchy that improve grading consistency [3][5].
