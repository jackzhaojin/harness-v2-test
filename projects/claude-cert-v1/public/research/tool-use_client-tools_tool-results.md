# Tool Results

**Topic ID:** tool-use.client-tools.tool-results
**Researched:** 2026-03-01T12:00:00Z

## Overview

Tool results are the mechanism by which your application returns execution outcomes back to Claude after processing a tool call. When Claude decides to use a tool, it emits a `tool_use` content block with a unique `id`, the tool `name`, and the `input` parameters [1]. Your application executes the corresponding function, then returns the result in a `tool_result` content block that references the original `tool_use_id` [1]. This creates a feedback loop where Claude can incorporate real-world data and external system responses into its reasoning.

The tool result mechanism is fundamental to Claude's agentic capabilities. Without proper tool result formatting, Claude cannot complete multi-step workflows, chain tool calls together, or synthesize information from external sources [1]. Understanding the exact structure, content types, and error handling patterns is critical for building robust tool-using applications.

## Key Concepts

- **tool_result content block** - A structured JSON object sent in a user message that contains the output of a tool execution. Must include `type: "tool_result"`, `tool_use_id`, and `content` [1].

- **tool_use_id** - A unique identifier that links a tool result to its originating tool call. This ID is generated by Claude in the `tool_use` block and must be matched exactly in the corresponding `tool_result` [1].

- **content field** - The actual result payload. Can be a simple string (`"15 degrees"`), an array of content blocks (`[{"type": "text", "text": "..."}]`), or document blocks [1][3].

- **is_error flag** - An optional boolean field set to `true` when tool execution fails. Signals to Claude that the result represents an error condition rather than successful output [1][2].

- **Multimodal tool results** - Tool results can include images and documents alongside text, enabling tools that return screenshots, charts, or file contents [1][4].

- **Parallel tool results** - When Claude makes multiple tool calls in one response, all corresponding tool results must be returned in a single user message [1].

- **Message ordering** - Tool result blocks must appear first in the content array of a user message, before any text blocks [1].

## Technical Details

### Basic tool_result Structure

The minimal tool_result requires three fields [1]:

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01A09q90qw90lq917835lq9",
      "content": "15 degrees"
    }
  ]
}
```

### Content Type Options

The `content` field supports multiple formats [1][3]:

**String content (simplest)**:
```json
"content": "San Francisco: 68F, partly cloudy"
```

**Array of text blocks**:
```json
"content": [
  {"type": "text", "text": "Temperature: 68F"},
  {"type": "text", "text": "Condition: Partly cloudy"}
]
```

**Image content** (for tools returning visual data) [1][4]:
```json
"content": [
  {"type": "text", "text": "Screenshot captured"},
  {
    "type": "image",
    "source": {
      "type": "base64",
      "media_type": "image/png",
      "data": "/9j/4AAQSkZJRg..."
    }
  }
]
```

**Document content** [3]:
```json
"content": [
  {
    "type": "document",
    "source": {
      "type": "text",
      "media_type": "text/plain",
      "data": "File contents here"
    }
  }
]
```

### Error Handling Format

When tool execution fails, set `is_error: true` and provide a descriptive error message [1][2]:

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01A09q90qw90lq917835lq9",
  "content": "ConnectionError: weather service API unavailable (HTTP 500)",
  "is_error": true
}
```

### Parallel Tool Results

When Claude requests multiple tools simultaneously, return all results in one user message [1]:

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01",
      "content": "San Francisco: 68F"
    },
    {
      "type": "tool_result",
      "tool_use_id": "toolu_02",
      "content": "New York: 45F"
    },
    {
      "type": "tool_result",
      "tool_use_id": "toolu_03",
      "content": "2:30 PM PST"
    }
  ]
}
```

### Empty Tool Results

For tools that perform actions without returning data (like sending an email), you can omit the content field [1]:

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01A09q90qw90lq917835lq9"
}
```

## Common Patterns

### Tool Call Loop Pattern

The standard pattern for handling tool calls follows this sequence [1]:

1. Send initial request with tool definitions
2. Check response `stop_reason` - if `tool_use`, extract tool calls
3. Execute each tool with the provided input
4. Format results as `tool_result` blocks matching `tool_use_id`s
5. Send new request with assistant message (tool_use) + user message (tool_results)
6. Repeat until `stop_reason` is `end_turn`

### SDK Tool Runner Pattern

Modern SDKs provide tool runners that automate the loop [1]:

```python
@beta_tool
def get_weather(location: str) -> str:
    """Get weather for a location."""
    return json.dumps({"temp": "68F", "condition": "sunny"})

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-6",
    tools=[get_weather],
    messages=[{"role": "user", "content": "Weather in SF?"}]
)
final = runner.until_done()
```

### Error Recovery Pattern

Provide actionable error messages that help Claude adapt [1][2]:

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_xyz",
  "content": "Rate limit exceeded. Retry after 60 seconds or use cached data.",
  "is_error": true
}
```

Claude typically retries 2-3 times with corrections before informing the user of persistent failures [1].

### Cache Control Pattern

Add cache control to large tool results for prompt caching [1]:

```python
for block in tool_response.content:
    if block.type == "tool_result":
        block.cache_control = {"type": "ephemeral"}
```

## Gotchas

- **Message ordering is strict** - Tool result blocks must come BEFORE any text in the content array. Placing text first causes a 400 error: "tool_use ids were found without tool_result blocks immediately after" [1].

- **All parallel results in one message** - When Claude makes multiple tool calls, you cannot send separate user messages for each result. All `tool_result` blocks must be in a single user message, or Claude will stop making parallel calls [1].

- **Never add text between tool_use and tool_result** - There must be no messages between Claude's tool_use response and your tool_result message. The tool result message must immediately follow [1].

- **Match tool_use_id exactly** - The `tool_use_id` must match the ID from Claude's request character-for-character. IDs look like `toolu_01A09q90qw90lq917835lq9` [1][3].

- **Return strings, not primitives** - Numbers, booleans, and other non-string primitives must be converted to strings. Return `"42"` not `42` [1].

- **Error messages should be instructive** - Avoid generic errors like "Failed". Include what went wrong and what Claude should try next: "Invalid location 'xyz'. Use format 'City, State' or 'City, Country'" [1][2].

- **Empty responses need no content** - For void operations, omitting `content` is valid. Do not return `"content": null` or `"content": ""` [1].

- **JSON must be stringified** - If returning structured JSON data, encode it as a JSON string. Claude will parse it [1].

- **Image size limits apply** - Images in tool results follow the same limits as regular image inputs: max 8000x8000 pixels, rejected if larger [4].

- **max_tokens truncation** - If Claude's response is cut off mid-tool_use due to max_tokens, you cannot return a tool_result for the incomplete block. Retry with higher max_tokens [1].
