# Thinking Redaction and Safety

**Topic ID:** safety-alignment.principles.thinking-redaction
**Researched:** 2025-03-01T12:00:00Z

## Overview

Thinking redaction is a safety mechanism in Claude's extended thinking feature that encrypts portions of the model's internal reasoning when flagged by Anthropic's safety systems. When Claude engages in extended thinking—a feature that allows the model to reason through complex problems step-by-step—its internal thought process may occasionally contain content that could be potentially harmful if exposed directly to users, including topics related to child safety, cyber attacks, or dangerous weapons.

When this occurs, Anthropic's safety systems automatically encrypt the flagged thinking content and return it as a `redacted_thinking` block instead of a standard `thinking` block. This approach allows Claude to continue using its full reasoning capabilities internally while preventing potentially harmful intermediate reasoning from being displayed. The encrypted content remains functional: when passed back to the API in subsequent requests, `redacted_thinking` blocks are decrypted server-side, allowing Claude to maintain reasoning continuity without losing context.

This mechanism represents a balance between transparency (showing users Claude's thought process) and safety (preventing exposure of dangerous reasoning). It applies specifically to Claude 3.7 Sonnet's extended thinking feature. Claude 4 models handle thinking differently through summarization and do not produce `redacted_thinking` blocks.

## Key Concepts

- **Extended Thinking**: A feature that gives Claude enhanced reasoning capabilities by allowing it to generate internal `thinking` content blocks before producing its final response. This improves performance on complex tasks requiring step-by-step analysis.

- **Thinking Block**: A content block of type `thinking` containing Claude's visible internal reasoning, accompanied by a `signature` field for verification. These blocks appear before `text` blocks in API responses.

- **Redacted Thinking Block**: A content block of type `redacted_thinking` containing encrypted reasoning content in a `data` field. Created when safety systems flag internal reasoning as potentially harmful.

- **Signature Field**: A cryptographic token attached to thinking blocks that verifies the content was generated by Claude. Required for the API to accept thinking blocks in subsequent requests. Signatures are cross-platform compatible (Claude API, Amazon Bedrock, Vertex AI).

- **Safety System Flagging**: Automated systems that monitor Claude's internal reasoning for potentially harmful content. Topics that trigger flagging include child safety, cyber attacks, dangerous weapons, and similar sensitive areas.

- **Summarized Thinking**: Claude 4 models return summaries of the thinking process rather than full thinking output, which eliminates the need for redacted thinking blocks while preventing misuse.

- **Thinking Block Preservation**: The requirement to pass complete, unmodified thinking blocks back to the API during tool use to maintain reasoning continuity.

## Technical Details

### API Response Structure

When extended thinking is enabled, responses contain thinking blocks followed by text blocks:

```json
{
  "content": [
    {
      "type": "thinking",
      "thinking": "Let me analyze this step by step...",
      "signature": "WaUjzkypQ2mUEVM36O2TxuC06KN8xyfbJwyem2dw3URve/op91XWHOEBLLqIOMfFG..."
    },
    {
      "type": "redacted_thinking",
      "data": "EmwKAhgBEgy3va3pzix/LafPsn4aDFIT2Xlxh0L5L8rLVyIwxtE3rAFBa8cr3qpP..."
    },
    {
      "type": "text",
      "text": "Based on my analysis..."
    }
  ]
}
```

### Enabling Extended Thinking

```python
response = client.messages.create(
    model="claude-3-7-sonnet-20250219",
    max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 10000},
    messages=[{"role": "user", "content": "Your prompt here"}]
)
```

### Handling Redacted Thinking in Multi-Turn Conversations

When using tool use with extended thinking, you must pass all thinking blocks (including redacted ones) back to the API:

```python
# Extract all thinking-related blocks
thinking_blocks = [
    block for block in response.content
    if block.type in ["thinking", "redacted_thinking"]
]

# Include them unmodified in the next request
continuation = client.messages.create(
    model="claude-3-7-sonnet-20250219",
    max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 10000},
    tools=[weather_tool],
    messages=[
        {"role": "user", "content": original_prompt},
        {"role": "assistant", "content": response.content},  # Full content including thinking
        {"role": "user", "content": [{"type": "tool_result", "tool_use_id": tool_id, "content": result}]}
    ]
)
```

## Common Patterns

### Detecting and Handling Redacted Thinking

```python
def process_response(response):
    has_redacted = any(block.type == "redacted_thinking" for block in response.content)

    if has_redacted:
        # Log for monitoring but continue normally
        log_info("Response contains redacted thinking")

    # Extract visible thinking for display
    visible_thinking = [
        block.thinking for block in response.content
        if block.type == "thinking"
    ]

    # Extract text response
    text_response = next(
        block.text for block in response.content
        if block.type == "text"
    )

    return {
        "thinking": visible_thinking,
        "response": text_response,
        "had_redaction": has_redacted
    }
```

### User-Facing Applications

When building applications that display thinking:

1. Filter redacted blocks from user view while preserving them for API calls
2. Provide explanatory UI: "Some internal reasoning was automatically encrypted for safety reasons"
3. Ensure redacted blocks don't break your UI rendering

### Preserving Blocks for Tool Use Loops

In multi-step tool use, maintain the full assistant content:

```python
messages = [initial_user_message]

while tool_use_in_progress:
    response = client.messages.create(...)

    # Store complete response content (including all thinking blocks)
    messages.append({"role": "assistant", "content": response.content})

    if has_tool_use(response):
        tool_result = execute_tool(response)
        messages.append({"role": "user", "content": [tool_result]})
    else:
        break
```

## Gotchas

- **Claude 3.7 vs Claude 4**: Redacted thinking blocks only occur in Claude 3.7 Sonnet. Claude 4 models use summarized thinking instead and never produce `redacted_thinking` blocks. Code written for Claude 3.7 may need updates when migrating.

- **Mandatory Block Preservation**: When using tool use with extended thinking on Claude 3.7 Sonnet, failing to pass back thinking blocks causes API errors: "Expected `thinking` or `redacted_thinking`, but found `text`." The API requires thinking blocks at the start of assistant turns.

- **Billing Surprise**: You are billed for full thinking tokens, not summary tokens. The visible token count in Claude 4 responses (summarized thinking) does not match billed output tokens.

- **Unmodified Blocks Only**: Thinking blocks must be passed back exactly as received. Any modification invalidates the signature and causes API rejection.

- **No Decryption for Users**: The `data` field in `redacted_thinking` blocks is not decryptable client-side. It's an opaque blob that only Anthropic's servers can process.

- **Cross-Platform Compatibility**: Signature values work across Claude API, Amazon Bedrock, and Vertex AI. However, the detailed encryption mechanism is not publicly documented.

- **Higher Refusal Rates**: Extended thinking can make Claude more cautious, particularly in areas like chemical weapons, cybersecurity, and human trafficking topics—even for legitimate questions.

- **Faithfulness Uncertainty**: Anthropic acknowledges that visible thinking may not perfectly represent internal model reasoning. Models may make decisions based on factors not explicitly shown in thinking output.

- **Cache Invalidation**: Changing thinking parameters (budget_tokens, enabled/disabled) invalidates cached message prefixes, though system prompts remain cached.
